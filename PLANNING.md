# VoidArchitect Engine & 4X Game Project Plan

## 1. Project Vision & Brief Description

This document outlines the planning for a hobbyist 4X space strategy game built using a custom engine named **VoidArchitect**.

The game aims to blend deep procedural generation inspired by _No Man's Sky_ (planets, stars, civilizations, cultures, history, ships, species, and more) with strategic management, diplomacy, and empire-building elements reminiscent of _Civilization_, _Anno_, _Cities Skylines_, and _Stellaris_.

The player will create a species and guide it from its tribal origins on a home planet through technological eras, eventually reaching the space age and interacting with a procedurally generated galaxy.

This is a long-term, solo-developer hobby project focused on learning and exploration of game development concepts, particularly in procedural generation and engine architecture. The development timeline is flexible ("Dwarf Fortress"-like).

## 2. High-Level Goals

- **VoidArchitect Engine:**
  - **Performant:** Leverage modern graphics APIs (starting with Vulkan) and parallel computation.
  - **Versatile:** Provide a solid foundation adaptable to different game systems.
  - **Procedural/Parallel Focused:** Built with capabilities for large-scale procedural generation and efficient parallel processing in mind.
- **Game:**
  - **Deep Simulation:** Simulate complex systems for civilizations, economies, ecosystems, etc.
  - **Vast Scale:** Procedurally generate a rich and expansive universe.
  - **Emergent Gameplay:** Foster interesting interactions through systemic design.
  - **Massively Multiplayer:** Design with client-server architecture to potentially support multiplayer and offload heavy simulation to a server.

## 3. Core Pillars

- **Extreme Procedural Generation:** Nearly every aspect of the game world should be generatable.
- **Strategic Depth:** Meaningful choices in management, diplomacy, research, and expansion.
- **Scalability & Performance:** Handling potentially vast amounts of data and simulation efficiently.
- **Modularity:** Designing the engine and game systems to be extendable.

## 4. Technical Architecture

- **Client-Server Model:**
  - **Server:** Handles the core simulation state, game logic, authoritative world state, potentially heavy procedural generation tasks, and manages client connections. Written in Rust.
  - **Client:** Responsible for rendering the game state received from the server, handling user input, and potentially some client-side prediction/interpolation. Written in Rust.
  - **Communication:** Uses TCP sockets for reliable transport. Data serialization via Protocol Buffers (Protobuf) using the `prost` library in Rust. UDP may be used for real-time updates.
  - **Shared Code:** A dedicated Rust crate (`void-architect_shared`) contains the Protobuf message definitions (`.proto` files) and the Rust code generated by `prost`. It's goal is to contain every const, struct, enum, and trait used by both the client and the server, with a minimum of dependencies. This crate is used as a dependency by both the client and the server to ensure consistency.

## 5. Technology Stack

- **Primary Language:** Rust (for both client and server)
- **Asynchronous Runtime:** Tokio (for network I/O and potentially other async tasks)
- **Networking:** TCP (via `tokio::net`), UDP (for real-time updates), Protobuf (via `prost` and `prost-build`)
- **Graphics API (Initial):** Vulkan (using `ash` or similar safe wrappers)
- **Windowing:** `SDL3` (cross-platform window creation and event handling)
- **Build System:** Cargo (Rust's build system and package manager)
- **Potential Future APIs:** OpenGL, Metal, DirectX (as engine knowledge expands)
- **AI/Procedural Generation:** Custom algorithms, potentially leveraging ML models or advanced techniques in the future.

## 6. Project Scope & Constraints

- **Solo Developer:** All development done by one person.
- **Hobby Project:** No commercial deadlines; focus is on learning and the process.
- **Long-Term:** Development may span several years.
- **Incremental Development:** Build features milestone by milestone, with each milestone leading to a working client and server.
- **Learning Focus:** Choice of technologies (Rust, Vulkan) driven partly by learning goals.

## 7. Initial Directory Structure

```
void-architect
├── assets
├── client
├── server
├── shared
├── engine_client
├── engine_server
└── core
```

- **Assets:** Contains images, models, and other resources.
- **Client:** The binary crate implementing the client, aka void-architect_client. Will be responsible for the game-specific logic.
- **Server:** The binary crate implementing the server, aka void-architect_server. Will implement the game-specific server-side logic.
- **Shared:** Shared code between client and server, aka void-architect_shared. Will contain messages protocol, shared structs, enums, and traits. Will be used by both the client and the server. Should have minimum dependencies.
- **Engine Client:** Lib crate implementing the client-side engine, aka void-architect_engine_client. Will implement client-side engine systems (rendering, input handling, etc.) that can be generalised and reused in other projects.
- **Engine Server:** Lib crate implementing the server-side engine, aka void-architect_engine_server. Will implement server-side engine systems (networking, game logic, etc.) that can be generalised and reused in other projects.
- **Core:** Lib crate implementing the engine core, aka void-architect_engine. Will implement the core engine systems which can be shared between client and server (ProcGen algorithms, math utilities, etc.).

## 8. Testing Architecture

### Test Organization

- **Unit Tests (`#[cfg(test)]` modules):**
  - Located alongside source code in the same file
  - Test pure functions, algorithms, and isolated components
  - Focus on validating individual function behavior
  - Validate invariants and edge cases
  - Test private implementation details

- **Integration Tests (`./tests/` directory):**
  - Test interaction between multiple components
  - Complex setup requirements or scenarios
  - End-to-end functionality tests
  - Performance benchmarks
  - User workflows and scenarios

### Specialized Testing Tools

- **Property-Based Testing (proptest):**
  - Used for testing functions with well-defined properties
  - Generates random inputs to validate invariants
  - Particularly useful for:
    - Math operations and algorithms
    - Data structure operations
    - Parsing and serialization
    - State machine transitions

- **Performance Benchmarking (criterion):**
  - Measure performance of critical operations
  - Track performance changes over time
  - Compare different implementations
  - Focus areas:
    - Procedural generation algorithms
    - Rendering operations
    - Network serialization/deserialization
    - Physics calculations

- **Mock Objects (mockall):**
  - Create test doubles for external dependencies
  - Simulate different scenarios and edge cases
  - Verify interaction patterns
  - Common uses:
    - Network operations
    - File system operations
    - Hardware interfaces
    - External services

### Testing Guidelines

1. **Coverage Goals:**
   - All public APIs must have tests
   - Critical paths require both unit and integration tests
   - Performance-sensitive code needs benchmarks
   - Edge cases and error conditions must be tested

2. **Test Organization:**
   - Group related tests in submodules
   - Use descriptive test names
   - Follow `test_<what>_<scenario>` naming convention
   - Keep test code clean and maintainable

3. **Best Practices:**
   - Write tests alongside new code
   - Update tests when modifying functionality
   - Use fixtures and helpers to reduce duplication
   - Document complex test setups
   - Include both positive and negative test cases

## 9. Defined Milestones

- **Milestone 0: Basic Console Communication (Client/Server)**
  - **Goal:** Establish a minimal, working TCP connection between a Rust client and Rust server. Send/receive simple "Hello" messages serialized with Protobuf. Use `tokio` for async networking. Implement the `void-architect_shared` crate.
  - **Status:** Done.
- **Milestone 1: Window & Basic Vulkan Setup**
  - **Goal:** Create a desktop window using `winit`. Initialize the Vulkan API: create an instance, select a suitable physical device, create a logical device, and potentially acquire a queue family. No rendering yet, just setup.
  - **Status:** Done.
- **Milestone 2: Static Vulkan Triangle**
  - **Goal:** Render a static triangle using Vulkan. Implement a basic rendering pipeline, handle vertex and index buffers, and draw calls.
  - **Status:** In Progress.
- **Milestone 3: Persistant Mobile Cube**
  - **Goal:** Integrate the network communication with the rendering, display a mobile cube that can be moved by the server, and rendered by the client. The state of the cube is persisted and synchronized between the client and the server.
  - **Status:** Planned (Epic and Stories defined).
- **Milestone 4: Basic Camera Control**
  - **Goal:** Implement basic camera controls (e.g., orbiting, panning, zooming) on the client-side, allowing the user to change their view of the scene (containing the cube from M3).
  - **Status:** To Plan.
- **Milestone 5: Basic Scene Representation**
  - **Goal:** Develop a rudimentary scene graph or entity management system on the client and server to handle multiple objects, not just a single cube. Synchronize the state of multiple objects.
  - **Status:** To Plan.
- **Milestone 6: Initial Procedural Noise Generation (Core)**
  - **Goal:** Implement basic procedural noise generation for terrain or other purposes.
  - **Status:** To Plan.

## 9. Future Considerations

- Developing more sophisticated procedural generation algorithms.
- Integrating potential AI/ML models for generation or NPC behavior.
- Designing a more robust and scalable multiplayer architecture.
